# v0.3 handoff brief — for the Agent Framework engineer driving live benchmarking

**Audience:** an engineer or applied researcher familiar with Microsoft
Agent Framework, Azure OpenAI quota allocation, and SWE-bench
Verified, who will take this repository and execute the four-arm
comparison the [v0.3 plan](v0.3-swe-bench-four-arm.md) describes.

**Why this brief exists:** the repo owner explicitly does not want an
LLM coding assistant making methodology calls (statistical power,
multi-seed variance, harness flakiness, comparison framing) that
belong to a benchmarking expert. v0.3 ships **infrastructure**, not a
**run**. This brief is the contract between the harness and the
person running it.

**What you should NOT assume from this brief:** that any of the
methodology choices below are correct. They are starting points to
push back on.

---

## 1. What is actually shipped, what is stubbed

### Shipped (works end-to-end, has tests)

- **Four-arm orchestration.** `bench/swe_bench/runner.py` runs every
  instance through all four arms (`router` / `single` / `pciv` /
  `fleet`) using the same `Router` the production CLI uses, and the
  same `--force-strategy` channel.
- **Failure-mode classification.** Five buckets
  (`budget-exceeded` / `iterate-exhausted` / `timeout` /
  `infra-error` / `reject`). See `_classify_failure()` in
  `bench/swe_bench/runner.py`. **Pre-classification: validate this maps
  cleanly onto the failure modes you care about. The current set is
  driven by what the strategies emit, not by what's interesting to
  report.**
- **Per-arm aggregation.** `summarise_arm()` computes resolved-rate,
  total cost, cost-per-resolved, p50/p95 latency and tokens, and the
  failure-mode histogram per arm.
- **Manifest with `git_sha` + `config_hash`.** `build_manifest()`
  records the SHA-256 of `config/policy.yaml` so a results file is
  bound to the policy that produced it.
- **30-instance frozen list** at
  `bench/swe_bench/instances_v0_3.txt` (seed 20260429, sorted IDs,
  immutable per `bench/swe_bench/README.md`).
- **Cassette redaction.** `bench/live/cassette.py:Cassette.save()`
  walks the payload through `agentcore.redaction.redact()` so live
  recordings cannot leak `OPENAI_API_KEY=...`, bearer tokens, or env
  literals. Five tests at `tests/test_cassette_redaction.py`.
- **Cross-run budget ledger.** `agentcore.budget.PersistentBudgetLedger`
  is wired into the strategies. Set `[cross_run].cap_usd` in
  `policy.yaml` to gate total live spend across processes; the ledger
  DB lives under `config/.budgeteer/`.
- **Stub-mode smoke run.** `just bench-smoke` (or
  `uv run python -m bench.swe_bench.runner --n 5`) produces a
  well-formed `bench/results/smoke/results.json` with all four arms,
  ~$0 spend, deterministic.

### Stubbed (NOT shipped, must be built before live run)

These are the **explicit gaps** between "harness compiles" and
"harness runs the v0.3 plan."

1. **Real `problem_statement` loading.** Stub mode passes
   `f"resolve {instance_id}"` as the task. Live needs the actual
   `problem_statement` field from
   `princeton-nlp/SWE-bench_Verified`. Suggested wiring: add
   `--problems-file` to the CLI, accept a JSONL of
   `{instance_id, problem_statement}` records, fail loudly if any
   ID in `instances_v0_3.txt` is missing.
2. **Azure OpenAI cassette adapter.** `bench/live/runner.py:run_live`
   raises `NotImplementedError` for `provider="azure_openai"`. Today
   only Anthropic is recorded. The cassette schema (AB-6, v1) is
   provider-agnostic; what's missing is an Azure-OpenAI-shaped
   adapter that satisfies `StreamingChatClient` and round-trips
   through the cassette.
3. **SWE-bench evaluation Docker pipeline.** None of: clone the
   repo, install deps, apply the diff, run the test harness inside
   the SWE-bench eval image. `PCIV/scripts/swe_bench_run.py` already
   does most of this for the PCIV-only path; lift the orchestration
   over (with citation) and have it consume the `StrategyResult.diff`
   from each arm. Today the harness just records what the strategy
   emitted; "resolved" in `results.json` is currently
   `result.success`, which is **strategy completion**, not
   **patch-passes-tests**.
4. **Real `RepoSnapshot` for routing decisions.** Stub mode uses
   `_empty_repo_snapshot()` (10 files, py, has_tests=True). Router
   decisions partly depend on the snapshot, so a live run must scan
   the cloned repo before invoking the router. Use
   `agentcore.scan.scan_repo()` (already a dependency).
5. **A live caller for the harness.** The current `main()` only
   knows stub mode. Add a `--live` flag that swaps in the real Azure
   OpenAI cassette adapter, the real PCIV subprocess runner, the
   real problem statements, and the SWE-bench Docker eval step.
   Keep stub mode as the CI smoke target; do not delete it.

## 2. Methodology questions to settle BEFORE you run

These are not rhetorical. The repo owner explicitly does not want the
LLM picking answers. These are listed in the order the answers will
shape the run.

### Q1. Sample size: is N=30 defensible?

The locked list is 30 instances, smallest defensible N for a 4-arm
comparison. **Concrete statistical concern:** at N=30, a single arm
flipping 1 instance moves resolved-rate by 3.3 percentage points.
A 5-point gap between arms is barely outside one-instance noise.
Options:

- Run all four arms over N=30 first, look at variance, decide whether
  to extend to N=100 (open `instances_v0_4.txt`).
- Run a power calculation against a target effect size before
  committing the budget.
- Pre-register a stopping rule.

### Q2. Stochastic-model variance: how many seeds per instance?

Azure OpenAI calls are not deterministic even at temperature=0. A
single pass per (arm, instance) gives a point estimate with unknown
variance. The brief mentions k=3 runs per instance for Bayesian
estimates. **Decisions you need to make:**

- k=1 (cheap, can't claim significance), k=3 (3× the budget, can fit
  a variance estimate per cell), or higher.
- One seed list shared across arms vs. per-arm seeds. (Shared lets
  you do paired comparisons, which are dramatically more powerful at
  small N.)
- How to aggregate: mean of k passes, majority vote on resolution,
  best-of-k, etc. **These are not equivalent comparisons.**

The harness today does **k=1**. It does not record a seed. Both need
fixing before live.

### Q3. Selection bias: 53% Django

The locked v0.3 list breaks down as 16 django / 8 sympy / 2 astropy /
2 scikit-learn / 1 pytest / 1 sphinx. That distribution is a property
of the underlying SWE-bench Verified set, but it means **any
arm-specific advantage on Django generalises with low confidence to
SWE-bench at large**, and **any per-repo failure mode in our
strategies (e.g. PCIV's planner choking on Django ORM patterns) will
be amplified**. Options:

- Report per-repo breakdown alongside the headline number.
- Stratify the next sample (`instances_v0_4.txt`) to deflate Django.
- Just disclose the skew in REPORT.md and let readers calibrate.

### Q4. Comparison framing: equal-budget, equal-time, or equal-effort?

The plan currently passes `--budget $X` to all four arms. That is an
**equal-budget** comparison. Other framings produce different stories:

- **Equal-time:** cap wall-clock per instance. Single-agent looks
  better; PCIV with critique loops gets penalised.
- **Equal-tokens:** cap input+output tokens. Fleet looks worse
  because shards multiply tokens.
- **Equal-effort (best at fixed $):** sweep the budget for each arm,
  draw the Pareto curve.

Pareto framing is what the thesis claims. Equal-budget gives you one
point per arm, not a curve. **If the headline is "router beats every
fixed strategy on the cost/quality Pareto frontier", the run needs
multiple budget points per arm (e.g. $0.50, $1.00, $2.50, $5.00).
That is a different, larger run than what the plan currently
describes.**

### Q5. Harness flakiness.

SWE-bench eval failures (Docker network blips, package install
flakes, OOMs in test runs) are not the strategy's fault but will
land in `infra-error`. You need a retry policy:

- Retry once on `infra-error`, fail closed otherwise?
- Retry the eval step but not the model call?
- Quarantine instances that flake in >1 arm and exclude them from the
  resolved-rate denominator?

Whichever you pick, **document it in REPORT.md before you start**.

### Q6. What gets reported.

The current `results.json` has per-instance records and per-arm
summary. REPORT.md does not exist. The brief asked for:

- Per-arm table (resolved-rate, $, $/resolved, p50/p95 latency,
  tokens).
- Pareto plot ($ vs. resolved-rate).
- Failure-mode breakdown.
- Methodology + limitations section.

For stochastic outcomes the Pareto plot needs uncertainty regions
(error bars, not points). Pick a method; the harness does not pick
one for you.

## 3. Competitive framing (per the original brief; do not skip)

The thesis is **"router beats fixed strategy on Pareto"**. That
statement only lands if the comparison set is honest. The brief
explicitly carves the field into two axes:

### Cost-routing axis: prior art the router is competing with

- **LiteLLM** — proxy + per-call routing on price/latency.
- **Portkey** — hosted gateway, similar shape.
- **OpenRouter** — model-agnostic API, automatic fallback.

These route **between models**. AgentBudgeteer's claim is that it
routes **between agent topologies** (single vs. plan-critique vs.
parallel fleet) conditioned on remaining budget. **REPORT.md must say
explicitly what these tools do and don't do, and where AgentBudgeteer's
claim is differentiated.** A reader who can't distinguish "router
picks gpt-4o vs gpt-4o-mini" from "router picks single-pass vs
plan-critique-implement-verify" will dismiss the work.

### SWE-bench axis: prior art on this benchmark

- **SWE-agent** (Princeton) — the original.
- **Aider** — pair-programming style.
- **OpenHands** (formerly OpenDevin) — agentic shell.
- **Agentless** — no-agent baseline that performs surprisingly well.

These are the comparison set for "did topology selection actually
help?" The honest framing is **not** "AgentBudgeteer router beats
SWE-agent" (we don't have an apples-to-apples run); it is **"forced
single vs forced PCIV vs forced fleet — does the router beat the
best fixed choice across this instance set?"**. That's what the four
arms measure. The external prior-art comparison is context, not the
headline.

### Explicitly NOT a comparison target

- **`imran-siddique/agentic-architecture`** — the brief flagged this
  as "do not compare to" because it's not in the same problem class.
  Do not include it in REPORT.md.

## 4. The smallest path from "harness compiles" to "live run complete"

Ordered, with rough proportionality of effort. **Do not interleave;
finish each step before starting the next.**

1. **Wire real problem statements.**
   - Add `--problems-file` to the CLI.
   - Pre-load all 30 problem statements; fail if any ID is missing.
   - Verify stub-mode smoke still passes.
2. **Build the Azure OpenAI cassette adapter.**
   - Mirror `budgeteer.adapters.anthropic_adapter` shape.
   - Reuse the AB-6 cassette schema (no schema change).
   - Get cassette-redaction tests passing for at least one Azure
     payload shape.
   - Round-trip a single hand-recorded cassette under `replay`.
3. **Wire the SWE-bench Docker eval pipeline.**
   - Lift orchestration from `PCIV/scripts/swe_bench_run.py`.
   - Replace `result.success` in `_result_to_record()` with the
     actual harness verdict.
   - Decide and document the retry policy (Q5 above).
4. **Decide N, k, budget points, comparison framing** (Q1-Q4 above).
   Get sign-off from whoever holds the budget.
5. **Run a 1-instance live dry-run** through all four arms to confirm
   total spend tracks the projection.
6. **Run the full N (with k > 1 if you've decided that).**
7. **Write REPORT.md.** Methodology section first, results second,
   limitations third, competitive framing fourth.

## 5. `results.json` schema (what the harness emits today)

```jsonc
{
  "manifest": {
    "run_uuid": "...",
    "git_sha": "...",
    "config_hash": "<sha256 of policy.yaml>",
    "instances_file": "bench/swe_bench/instances_v0_3.txt",
    "policy_file": "config/policy.yaml",
    "n_instances": 30,
    "arms": ["router", "single", "pciv", "fleet"],
    "started_at": "...",
    "finished_at": "...",
    "mode": "stub",          // or "live"
    "schema_version": 1
  },
  "arms": {
    "router": {
      "summary": {
        "n": 30, "resolved": 0, "resolved_rate": 0.0,
        "total_cost_usd": 0.0, "cost_per_resolved_usd": null,
        "p50_latency_seconds": 0.0, "p95_latency_seconds": 0.0,
        "p50_tokens_in": 0, "p95_tokens_in": 0,
        "p50_tokens_out": 0, "p95_tokens_out": 0,
        "failure_modes": {
          "budget-exceeded": 0, "iterate-exhausted": 0,
          "timeout": 0, "infra-error": 0, "reject": 0
        }
      },
      "records": [
        {
          "instance_id": "django__django-12345",
          "arm": "router",
          "decision_strategy": "single|pciv|fleet",
          "decision_model": "azure-reasoning",
          "success": false,
          "cost_usd": 0.0,
          "tokens_in": 0,
          "tokens_out": 0,
          "latency_seconds": 0.0,
          "failure_mode": "infra-error",
          "error": "..."
        }
      ]
    },
    "single": { /* same shape */ },
    "pciv":   { /* same shape */ },
    "fleet":  { /* same shape */ }
  }
}
```

**`schema_version: 1`** is set at write-time. If you change the
schema, bump it and update `tests/test_swe_bench_runner.py`. The
replay path (when it exists) will key off this version.
